# Bootstrap Parallelization And Hardware Stress Summary

Generated by:

```powershell
npm.cmd run experiment:bootstrap-hardware
```

Environment:

- Windows_NT 10.0.26200 x64
- AMD Ryzen 9 8940HX, 32 logical CPUs
- 31.2 GiB RAM
- Node v24.13.0

## What Was Measured

Synthetic SBS96-like bootstrap workloads were run through the native JavaScript path using Node worker threads. These runs measure the bootstrap computation path, not browser UI rendering or a full notebook session.

## Main Results

| Scenario | Median Runtime | Median Peak RSS | Notes |
|---|---:|---:|---|
| Single sample, 500 iterations, serial | 1.510 s | 116 MiB | 67 signatures x 96 contexts |
| Single sample, 500 iterations, 4 workers | 0.845 s | 259 MiB | Faster wall-clock with higher memory |
| 120 samples, 25 iterations/sample, serial | 12.276 s | 235 MiB | 3,000 bootstrap fits |
| 120 samples, 25 iterations/sample, 4 workers | 4.647 s | 546 MiB | 3,000 bootstrap fits |
| 300 samples, 10 iterations/sample, 4 workers | 1.990 s | 453 MiB | 40 signatures x 96 contexts |

Heap-cap checks passed at 128 MiB for the single-sample 500-iteration two-worker test and 256 MiB for the 120-sample x 25-iteration four-worker test. These are Node heap-cap lower bounds for synthetic workloads, not universal browser/system-RAM requirements.

## Interpretation

The binding constraint for all-sample bootstrap is local compute, especially available CPU workers and the configured number of bootstrap refits. Public APIs are not involved once matrices and catalogs are loaded.

For manuscript language, describe the hardware guidance conservatively: a modern multi-core desktop or laptop browser is appropriate for cohort-scale review, and larger cohorts should tune bootstrap iterations or defer uncertainty to selected samples when interactive latency matters.
